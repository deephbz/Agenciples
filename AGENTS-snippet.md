## Work OS Principles

Five principles govern all research and engineering work here, because agents made implementation cheap and left concept clarity, investigability, and continuity as the scarce resources. (1) **Full-stack traceability**: important computations must be investigable across abstraction layers — clean default output plus a trace/debug mode exposing intermediate state and provenance, with raw records kept as source of truth and every important claim anchored to an external verification signal (a trace or doc alone is not proof of correctness). (2) **Ontology-first design**: identify the domain's irreducible nouns, verbs, and relations before designing APIs or generating code; encode distinctions in types, not prose; a good ontology yields a minimal public surface with few semantically stable verbs, schemas limited to the coordinates required for safe composition, and rich task meaning carried in agent-readable prose; interfaces preserve human intent while implementations are the agents' search space, so review interfaces harder than implementations. (3) **Artifact-first, backend-first**: work starts from a durable written problem artifact and ends with a durable result artifact whose canonical form is machine-operable (data + config + code version + interpretation) with visuals as derived renderings; version-control the full source bundle (code, configs, prompts, plans, evaluators — natural-language instructions needed for reproduction are source), persist final artifacts, treat intermediates as disposable, and expose reasoning/data lineage as reviewable graphs; idea/data diagrams are authored and agreed before development and refreshed at each milestone (alignment devices for humans, teams, and agents — not after-the-fact documentation), and each project keeps a curated evergreen doc (decisions still in force, current status, blockers, next steps) separate from append-only journals — never let results live only in chat, screenshots, or one-off notebooks. (4) **Single source of truth, fluid representation**: every fact, spec, or procedure has exactly one authoritative home, which migrates as work matures — shaping-stage truth lives in docs and diagrams; once interfaces stabilize, specs move into types and public APIs and docs trim to intent, rationale, and pointers; natural and programming language read the same to agents but differ in verifiability and reliability, so allocate deliberately and re-allocate as procedures harden (SOP doc → script → library), keeping docs and code pointing at each other. (5) **Stage-calibrated rigor ("where are we?")**: establish the project's lifecycle stage (shaping, exploration, consolidation, hardening, sharing) before choosing practices — infer it, ask when ambiguous, declare it in the evergreen doc; exploration wants glue scripts and fast insight to discover the right shape, extensive unit tests before the shape exists are tautological, and linters/formatters/CI/CD earn their keep only once work is functional, performant, and heading for a PR — stage lowers the formality of verification anchors, never their existence. Preserve three distinct information classes: historical evidence preserves source records and observations, which may be incomplete or wrong; current working context states what still matters and is corrected or superseded as understanding changes; assessments and projections state what we currently infer and must remain recomputable with provenance, freshness, and uncertainty. Scenario playbooks (module design, domain modeling, research artifacts, agent continuity, source allocation, stage calibration) live in the `work-os` skill; load it before non-trivial design, debugging, research, or agent-workflow work.
